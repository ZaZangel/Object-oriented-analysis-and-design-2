1. Введение
1.1. Цель работы
Изучить паттерн проектирования «Абстрактная фабрика» (Abstract Factory).
Реализовать паттерн на языке C# в среде Windows Forms.
Сравнить реализацию с паттерном и без него.
Продемонстрировать преимущества паттерна для расширяемости системы.
1.2. Задание (вариант 3.2)
Информационная система предназначена для «сборки» моделей автомобилей.
Модель автомобиля может включать компоненты: кузов, двигатель, коробка передач, диски/шины.
Описать каждый компонент классом с подходящим набором атрибутов.
Реализовать «сборку» различных моделей выбранного автопроизводителя (не менее 3) на основе реального каталога.
2. Теоретическая часть
2.1. Что такое паттерн Abstract Factory?
Абстрактная фабрика — порождающий паттерн проектирования, который предоставляет интерфейс для создания семейств взаимосвязанных или взаимозависимых объектов, не специфицируя их конкретных классов.
Ключевые участники паттерна:
Участник
Назначение
Пример в задаче
AbstractFactory
Объявляет интерфейс для создания абстрактных продуктов
CarFactory
ConcreteFactory
Реализует создание конкретных продуктов
BMWFactory, MercedesFactory, AudiFactory
AbstractProduct
Объявляет интерфейс для типа продукта
Body, Engine, Transmission, Wheels
ConcreteProduct
Реализует конкретный продукт
BMWBody, MercedesEngine и т.д.
Client
Использует только абстрактные интерфейсы
Form1, CarBuilder

4. Практическая реализация
3.1. Структура проекта
Проект организован в виде приложения Windows Forms со следующей структурой файлов:
Program.cs — точка входа в приложение
Form1.cs — основная форма с реализацией паттерна Abstract Factory
Form1.Designer.cs — дизайнер формы Form1
Form2.cs — форма сравнения без использования паттерна
Form2.Designer.cs — дизайнер формы Form2
CarFactoryApp.csproj — файл проекта

3.2. Реализация абстрактных продуктов
В рамках реализации паттерна определены четыре абстрактных класса, представляющих компоненты автомобиля:
Body (кузов) — содержит свойства Type, Material, Color
Engine (двигатель) — содержит свойства Type, Horsepower, FuelType
Transmission (коробка передач) — содержит свойства Type, Gears
Wheels (колёса) — содержит свойства Brand, Diameter, TireType
Каждый абстрактный класс объявляет интерфейс для соответствующего типа продукта.
3.3. Реализация конкретных продуктов
Для каждого производителя автомобилей реализованы конкретные классы продуктов:
BMW:
BMWBody, BMWEngine, BMWTransmission, BMWWheels
Mercedes-Benz:
MercedesBody, MercedesEngine, MercedesTransmission, MercedesWheels
Audi:
AudiBody, AudiEngine, AudiTransmission, AudiWheels
Каждый конкретный класс наследуется от соответствующего абстрактного класса и реализует его свойства с конкретными значениями характеристик.

3.4. Абстрактная и конкретные фабрики
Абстрактная фабрика CarFactory объявляет интерфейс для создания всех типов продуктов:
CreateBody() — создание кузова
CreateEngine() — создание двигателя
CreateTransmission() — создание коробки передач
CreateWheels() — создание колёс
ManufacturerName — свойство для получения названия производителя
Конкретные фабрики реализуют создание продуктов для конкретных производителей:
BMWFactory — создаёт компоненты BMW
MercedesFactory — создаёт компоненты Mercedes-Benz
AudiFactory — создаёт компоненты Audi
3.5. Класс автомобиля и сборщик
Класс Car представляет собранный автомобиль и содержит:
Свойства для всех компонентов (Body, Engine, Transmission, Wheels)
Свойство Model для названия модели
Метод GetDescription() для формирования описания автомобиля
Класс CarBuilder отвечает за сборку автомобиля:
Принимает фабрику в конструкторе
Метод BuildCar() создаёт автомобиль, запрашивая компоненты у фабрики
Инкапсулирует процесс сборки от клиента
4. Пользовательский интерфейс

Ключевое отличие в реализации:
В форме Form2 создание компонентов реализовано через условные операторы if/else без использования паттерна. Для каждого производителя данные создаются напрямую в виде объектов класса CarComponent.
Проблема: для добавления нового бренда необходимо изменять существующий код, что нарушает принцип Open/Closed.
5. Генерация HTML-отчёта
5.1. Класс HtmlGenerator
Для выполнения требования задания реализован статический класс HtmlGenerator, который:
Принимает список автомобилей и путь к файлу
Формирует HTML-разметку с использованием StringBuilder
Создаёт стилизованные карточки для каждого автомобиля
Записывает результат в файл с кодировкой UTF-8

6. Сравнительный анализ

6.1. Выводы по сравнению
Реализация с паттерном Abstract Factory требует больше классов и начальных усилий, но обеспечивает:
Гибкость: добавление нового производителя не требует изменения существующего кода
Надёжность: компилятор контролирует, что все компоненты созданы
Поддерживаемость: каждый класс отвечает за одну задачу
Реализация без паттерна подходит для прототипов, но становится проблемной при росте системы.
