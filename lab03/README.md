## Часть 1: Теория паттернов (по материалам книги GoF)

### Что такое паттерн проектирования?

**Паттерн (шаблон)** — это обобщённое решение часто встречающейся проблемы проектирования в определённом контексте.

> Паттерн — **не готовый код**, а **описание подхода**, который можно адаптировать под вашу задачу.

### Структура описания паттерна

| Элемент | Описание |
|---------|----------|
| **Имя** | Ключевое слово для быстрой коммуникации («Стратегия», «Наблюдатель») |
| **Проблема** | Когда применять паттерн, какие симптомы в коде указывают на его необходимость |
| **Решение** | Абстрактное описание классов, объектов и их взаимодействий |
| **Последствия** | Плюсы и минусы применения, компромиссы |

---

### Классификация паттернов GoF

Паттерны классифицируются по **двум параметрам**:

#### 1. По цели (назначению)

| Тип | Решает проблему... | Примеры |
|-----|-------------------|---------|
| **Порождающие** | Создания объектов | Фабричный метод, Абстрактная фабрика, Строитель, Прототип, Одиночка |
| **Структурные** | Композиции классов и объектов | Адаптер, Декоратор, Фасад, Компоновщик, Мост, Заместитель, Приспособленец |
| **Поведенческие** | Взаимодействия объектов | Итератор, Стратегия, Команда, Наблюдатель, Состояние, Посетитель, Посредник, Цепочка обязанностей, Хранитель, Шаблонный метод, Интерпретатор |

#### 2. По уровню применимости

| Уровень | Применим к... | Характеристика |
|---------|--------------|----------------|
| **Классы** | Отношениям между классами и подклассами | Статичны, фиксируются на этапе компиляции через наследование |
| **Объекты** | Отношениям между объектами | Динамичны, могут изменяться во время выполнения через композицию |

> Почти все паттерны используют наследование, но к «классовым» относят только те, что фокусируются **исключительно** на отношениях классов.

### Ключевые принципы, лежащие в основе паттернов

```text
┌─────────────────────────────────────────┐
│ 1. Инкапсуляция вариаций                │
│    → Выделяй то, что меняется, в отдельный объект │
├─────────────────────────────────────────┤
│ 2. Программируй на уровне интерфейсов   │
│    → Зависимость от абстракций, а не реализаций │
├─────────────────────────────────────────┤
│ 3. Композиция предпочтительнее наследования │
│    → Гибкость: можно менять поведение на лету │
├─────────────────────────────────────────┤
│ 4. Принцип открытости/закрытости (OCP)  │
│    → Открыто для расширения, закрыто для модификации │
├─────────────────────────────────────────┤
│ 5. Инверсия зависимостей (DIP)          │
│    → Модули высокого уровня не зависят от деталей │
└─────────────────────────────────────────┘
```

### Когда НЕ стоит применять паттерны?

- Когда не требуется повторное использование кода
- Когда система стабильна и не будет изменяться
- Когда паттерн усложняет код без реальной пользы («оверинжиниринг»)
- Когда задача решается проще без абстракций

> **Правило**: «Не применяй паттерн, пока не почувствуешь боль от его отсутствия».

### О чём этот код?

Это **демонстрационное веб-приложение CookBook Pro**, которое показывает на практике применение **паттерна Итератор (Iterator Pattern)** из группы поведенческих паттернов.

#### Цель проекта
Показать разницу между:
- **Наивной реализацией**: логика обхода коллекции «зашита» внутрь класса `Cookbook`
- **Реализацией с паттерном**: логика обхода вынесена в отдельные классы-итераторы

#### Что делает приложение?
Позволяет пользователю:
1. Просматривать коллекцию рецептов
2. Переключаться между режимами обхода:
   - `chronological` — в порядке добавления
   - `by_difficulty` — от лёгкого к сложному
   - `quick_first` — от быстрого к долгому
3. Переходить к следующему/предыдущему рецепту
4. Запускать анимацию «готовим»
5. Сбрасывать позицию на начало

### Ключевые фрагменты кода (бэкенд на Python)

#### 1. Абстракция итератора
```python
from abc import ABC, abstractmethod

class RecipeIterator(ABC):
    @abstractmethod
    def has_next(self) -> bool:
        pass
    
    @abstractmethod
    def next(self) -> Recipe:
        pass
    
    @abstractmethod
    def reset(self):
        pass
```

#### 2. Конкретный итератор (пример)
```python
class DifficultyIterator(RecipeIterator):
    DIFF_ORDER = {"easy": 0, "medium": 1, "hard": 2}
    
    def __init__(self, recipes: list[Recipe]):
        # Сортируем при создании — логика инкапсулирована
        self._sorted = sorted(recipes, 
            key=lambda r: (self.DIFF_ORDER.get(r.difficulty, 99), r.title))
        self._index = 0
    
    def next(self) -> Recipe:
        if self.has_next():
            self._index += 1
        return self._sorted[self._index]
```

#### 3. Фабричный метод в коллекции
```python
class Cookbook(RecipeCollection):
    def create_iterator(self, mode: str) -> RecipeIterator:
        if mode == "by_difficulty":
            return DifficultyIterator(self._recipes)  # ← Полиморфизм!
        elif mode == "quick_first":
            return QuickFirstIterator(self._recipes)
        # ...
```

#### 4. Клиентский код (не зависит от конкретики)
```python
# Переменная имеет тип абстракции!
current_iterator: RecipeIterator = cookbook.create_iterator("chronological")

# Вызовы работают с любым итератором — полиморфизм в действии
current_iterator.next()
current_iterator.has_next()
```

### Сравнение: с паттерном / без паттерна

| Критерий | Без паттерна | С Iterator Pattern |
|----------|-------------|-------------------|
| **Ответственность класса** | `Cookbook` делает всё: хранит + сортирует + обходит | `Cookbook` только хранит, итераторы — обходят |
| **Добавление нового режима** | Править метод `_get_sorted_list()` внутри `Cookbook` | Создать новый класс-итератор + добавить ветку в фабрику |
| **Тестируемость** | Сложно тестировать монолитную логику | Каждый итератор тестируется изолированно |
| **Повторное использование** | Низкое (логика привязана к коллекции) | Высокое (итераторы независимы) |
| **Принцип OCP** | Нарушен (модификация при расширении) | Соблюден (расширение без модификации) |
